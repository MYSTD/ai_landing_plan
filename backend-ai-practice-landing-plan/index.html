技术部门 AI落地实践与方案

AI 编程工具正在进入研发生产场景

2025 年开始，AI 作为编程工具逐渐引入到企业内部的生产环境中。到 2025 年 11 月，随着模型升级和发布，越来越多的人开始认可 AI 已经能在生产中发挥较大的作用。2026 年至今，随着模型进一步更新，模型的理解能力和编程能力进一步增强。

随之而来的是开发者对使用 AI 编程工具的观点也开始分化：有人坚持排斥，有人谨慎接受，也有人已经把 AI 当作提升生产力的主力工具。

根据我们目前的实践，我们当前处在半维新派和维新派之间，我们是切实感受到ai带来的能力和效率提升的

为什么不是维新派： 极致的AI使用成本花费巨大，需要用明确的目标和方向。

使用 AI 前后的后端研发流程对比

研发阶段 使用 AI 前 使用 AI 后 实践成果 / 目标 / 价值
需求文档审查 研发人员阅读产品需求，通过评审和沟通确认业务规则、实现边界及待明确事项。 AI 阅读文档，明确业务逻辑，转化成业务名词和专用术语，并辅助做规范性、合理性审查。 已用于日常产品需求文档审查；目标是统一审查文档规范性和合理性，对齐业务边界，减少错误理解和分歧。
需求文档转化为技术方案和技术文档 技术人员理解需求文档，编写技术文档，在开发前梳理变更点以及开发的核心逻辑。 AI 将需求文档转化成技术语言版本的需求文档，例如流程图、改动点、核心业务逻辑等，为开发做好准备。 已用于果币重构 + 退款功能中的复杂业务梳理；目标是基于历史数据和原有技术实现快速生成初版技术文档，加快开发对需求的理解，提升开发前准备效率。
技术实现设计、调研与风险评估 开发根据需求文档和技术目标进行技术方案设计、调研和风险评估。 开发提供系统背景和技术约束，AI 辅助生成方案、分析影响范围、补充风险点和异常场景，再由开发沟通并最终决策。 简单需求已能较快产出技术实现和风险评估，技术调研速度有明显提升；复杂项目仍由人重点把控，目标是提升方案合理性和产出效率。
编码实现 开发根据技术方案和文档编写代码。 AI 根据已完善的技术方案和文档编写代码，开发审核代码并确定实现的合理性和有效性。 已用于日常编码工作；目标是减少重复编码和样板代码耗时，提升编码效率，同时保留开发对实现质量的判断。
单元测试 工程师根据实现逻辑编写单元测试、测试数据和异常场景，并运行测试验证代码。 AI 辅助生成单元测试、模拟数据、边界场景和异常路径，工程师负责确认测试是否符合真实需求。 原有单测覆盖率不足 30%，AI 介入后目标提升至 80% 以上；覆盖率提升可以有效降低 bug 率，也能减少测试用例设计和补测成本，提高测试工作效率。
Code Review 审查人员逐行检查代码规范、实现逻辑、异常处理、性能和可维护性。 AI 先进行 Review 并覆盖全部代码变更，生成 Review 报告后再由人工复核。 目标是推动 Review 流程自动化并生成标准化 Review 报告；代码规范性和非复杂逻辑先由 AI 提高 Review 效率，人工重点复核复杂逻辑和业务判断。
问题排查 工程师复现问题、查看日志、跟踪程序运行过程，并分析响应速度、资源使用和数据处理瓶颈。 AI 结合日志、错误信息、代码和性能数据，辅助分析问题原因并给出修复或优化建议。 已通过AI进行问题分析，提高技术文档查阅和排查的速度

后端的AI选型

后端使用过的AI模型如下： Deepseek,千问，智谱GLM, MiniMax, Codex(gpt)， Grok(马斯克）， Gemini

好模型和差模型，在使用体验上的差异体现在：

  1. 能更好的预测你想要什么（AI本质上是一个预测模型）
  2. 在复杂项目的沟通中犯错更少，且始终聚焦目标。
  3. 更少的幻觉和以及纠正幻觉的准确率。
  4. 成本合理

这些差异会非常影响使用ai的效率。举例来说，和ai沟通一个改动，一次改完，ai的思考和实施时间可能是3分钟，但如果有问题，要再沟通进行调整，时间可能就是6分钟，时间代价加倍了。

近期国产ai的能力虽然有提升，但在编程方面的工作上的能力和Codex相比仍有差距，另外一个我们关注的点则是成本。以Codex为例，Codex团队包月的一个名额，费用在30美元，国产比如阿里云的Qoder则是300元。

综上我们目前考虑使用国外的模型来承载我们的开发工作。当然这不是说Codex一定是很长期的方案，如果模型的能力和价格发生变化，我们也会进行评估和调整

基于综合能力和编程能力的考量，chatgpt和Claude在模型能力方面都是达标的，但Claude 在价格方面并不友好，所以我们考虑的选型方案是chatgpt

落地方案：高额度账号采购 + 中间站分发

1、高额度账号的单价更便宜， 用量受到的限制更小

2、中间站统一分发

通过中间站将账号额度分发给不同的用户。目前以 ChatGPT/Codex 为主，后续扩展国内模型时不需要重新改变员工使用入口。

账号池首期统一接入 ChatGPT/Codex 账号，按状态、额度和可用性进行调度。
渠道池支持接入 OpenAI 接口格式的 API Key，例如 DeepSeek V4、GLM-5.1 等国内模型或其他兼容接口服务。
Key 分发按部门、小组或个人发放独立 API Key，员工仍使用统一 URL 和个人 Key。
额度/IP支持单 Key 启停、额度上限、IP 限制和后续动态调整。
用量依据按 Key 查看请求、Token 和消耗趋势，为加账号、调额度或接新渠道提供依据。
上线路径

四步完成内部接入

1. 中间站服务器海外低配云服务器一台吗，预计10$/月
2. 账号准备 2-3 个 ChatGPT Pro 账号。公开价格口径约 $200/账号/月，首批账号预算约 $400-$600/月。
3. 号池/渠道池把 ChatGPT 账号先导入 Sub2API，后续也可接入 OpenAI 格式的 API Key 渠道，由平台统一调度、统计和分发。
4. 使用员工在 Codex/Cockpit 中填写统一 URL 和个人 Key。后续按实际消耗调整人员或部门额度。
采购路径

采购方式与风险口径

账号来源会影响接入速度、成本和后续维护。考虑到当前需要尽快落地，建议首期优先采用现有相对稳定的第三方代充渠道，先把账号接入 Sub2API 统一分发；官方直购可作为后续长期路径继续评估。

官方直购稳定性和账号归属最清晰，但通常需要海外银行卡、海外账单地址和可用网络环境，前期办理周期和沟通成本较高。
第三方代充落地速度较快，适合首期优先采用。目前已有相对稳定的渠道，重点是把采购、续费和异常处理放到统一管理口径内，同时保留价格波动和账号风控的预案。
风险说明ChatGPT 不属于面向中国大陆开放使用的官方服务，访问和使用存在一定不确定性。内部自用、统一入口、限制范围的风险口径，可参考公司对海外 VPN 或海外 SaaS 工具的管理方式处理。

正式采购前以 OpenAI 官方支持地区 和公司内部合规意见为准，首期按内部研发工具管理，不做公众开放或对外销售。
